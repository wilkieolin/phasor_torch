# HPO summary — LCA vs LSA (audio keyword-spotting, ResonantSTFT frontend)

Everything here is generated by `scripts/analyze_hpo.py` (surrogate sweeps + incumbent tables)
and `scripts/eval_confirm_curves.py` (full-data confirmation train/val curves). Data sources:
`hpo_runs/{lca,lsa}/results.csv` (200 ytopt evals each, 16k subset) and the full-data confirm
PBS logs `phasor_confirm.o8555898` (LCA) / `phasor_confirm_lsa.o8602854` (LSA) + the checkpoints
in `hpo_confirm/{lca,lsa}/`.

## 1. Surrogate sweeps (16k subset, 200 evals, objective = -best_test_acc)

- **LCA subset peak = 66.3%** (found at eval 115); **LSA subset peak = 63.9%** (found at eval 186).
- LCA's incumbent climbs steadily and **plateaus by ~eval 115**, then a dense cloud of strong
  points confirms a broad good region.
- LSA gets **stuck at ~44% from eval 62–140**, then finds its real basin **late (eval 186)** — the
  proxy budget barely surfaced it. See §3 for why: LSA's winning config wants a *low* lr that the
  sweep only stumbled onto near the end.

## 2. Full-data confirmation (top-8 configs re-trained on full ~51k data)

The 16k subset is a good *filter* but a poor *ranker* — always confirm top-K, not top-1.

| | subset peak | **full-data peak** | full winner was subset rank |
|---|---|---|---|
| **LCA** | 66.3% | **78.1%** | rank **5** (subset-6th!) — ranking fully scrambled |
| **LSA** | 63.9% | **63.3%** | rank **0** (subset-1st) — top survived, rest reshuffled |

- **Full data widens the LCA→LSA gap from ~2.4 pt (subset) to ~15 pt** — LCA is decisively the
  better architecture at scale.
- **Overfit / peak-then-decline is real** on both. LCA confirm ran *pre-fix* (early-stop on
  test_loss, no restore) so its `checkpoint.h5` sits on the post-peak tail (up to −9 pt below peak,
  e.g. rank0 73.7→64.6). LSA confirm ran *post-fix* (early-stop on **test_acc** + **restore-best**),
  so its deployed `best.h5` = the peak, and it stopped 6–37 epochs earlier (20–42 vs configured
  48–79), skipping the overfit tail. **Deploy/report off `best.h5`.**

## 3. Divergent optima

- **LCA** wants **low n_heads (2)**, large **n_anchors (256)**, moderate **d_hidden (128)**,
  **lr ≈ 9e-4** (narrow band), **init_scale ≈ 2**. Interior/converged optimum.
- **LSA** subset optimum pinned to the **search boundary** (d_hidden=256, n_heads=8, lr up to
  6.7e-3). Full data disproves it: **full-data accuracy anti-correlates with lr** across the top-8
  (lowest lr rank0 1.09e-3 → 63.3%; highest rank5 6.74e-3 → 37.5%). Real full-data LSA wants
  **low lr (~1e-3)** — the boundary-pinned high-lr optimum was clipped/misleading.

## Reproduce & caveats

```bash
# surrogate scatter + incumbent tables (fast, reads results.csv only)
python scripts/analyze_hpo.py
# confirm curves: evaluate saved checkpoints on local data (slow, GPU ~1.5h) then plot
python scripts/eval_confirm_curves.py        # -> confirm_curves_data.json
python scripts/plot_confirm_from_json.py     # -> confirm_curves_eval_{arch}.png
```

- Confirm curves are sampled on the **every-10-epoch** saved checkpoints (`ckpt_epoch*.h5`)
  plus the peak (`best.h5`) and deployed final (`checkpoint.h5`). The dashed **peak-val line**
  can sit above the plotted markers when the true peak epoch falls between saved checkpoints
  (e.g. LCA rank0 peaks ~ep45; grid has ep40/ep50). Each rank's numbers were validated against
  the confirm PBS-log summary line (exact match).
- **This sparse reconstruction is only needed for the existing runs**, which predate the
  `confirm.py` history.json fix. Newer confirm runs write a full per-epoch `history.json` per
  rank — plot those directly, no checkpoint re-eval.

## Takeaways

1. **Ship LCA** for this task (78% vs 63% at full data).
2. **Confirm top-K on full data** — subset rank ≠ full rank (extreme for LCA).
3. **Watch lr for LSA** — widen *downward*, not upward; the ceiling was a trap.
4. **The test_acc early-stop + restore-best fix works** (LSA confirm) — captures the peak and cuts
   compute. Re-running the LCA confirm under the fix would recover its ~+8 pt tail loss automatically.
